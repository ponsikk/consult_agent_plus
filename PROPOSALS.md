# Архитектурные предложения по развитию проекта «Цифровой Инспектор»

Этот документ содержит стратегические предложения по улучшению архитектуры, UX и надежности системы для обсуждения с командой агентов (Claude/Gemini).

---

## 1. Автономность и работа в поле (PWA / Offline-First)
**Проблема:** Инспекторы часто работают на объектах с плохой связью (подвалы, стройплощадки). Текущий фронтенд при потере сети просто выдает ошибку.

**Предложение:**
*   **PWA (Progressive Web App):** Настроить Service Workers для кэширования статики и возможности запуска приложения без интернета.
*   **IndexedDB Storage:** Реализовать временное хранение фотографий и метаданных прямо в браузере. Юзер нажимает «Загрузить», фото уходит в локальную БД, и как только сеть восстанавливается — фоновая синхронизация (Background Sync) отправляет их на бэкенд.
*   **UI-индикация:** Добавить статус-бар «В сети / Офлайн» и очередь исходящих загрузок.

## 2. Самообучающийся цикл и RAG (Retrieval-Augmented Generation)
**Концепция:** Превращение системы из «простого интерфейса к нейронке» в экспертную базу знаний, которая учится на опыте компании.

### 2.1. Юзерский флоу (Human-in-the-Loop)
1.  **Первичный анализ:** Юзер загружает фото. ИИ выдает «сырой» черновик.
2.  **Режим правки (Review):** Инспектор в интерактивном редакторе правит рамки (bbox), описания, ссылки на ГОСТы и критичность.
3.  **Фиксация опыта:** Кнопка «Утвердить как эталон» сохраняет фото и идеальный JSON в таблицу `gold_standards`.

### 2.2. Технический флоу (OpenRouter RAG)
1.  **Retrieval:** Перед анализом нового фото система ищет 2-3 релевантных эталона по типу системы (например, 'roof_flat') или через векторный поиск по картинке.
2.  **Augmentation:** Бэкенд собирает Few-shot промпт: [Эталон 1 + Ответ человека] -> [Эталон 2 + Ответ человека] -> [Новое фото юзера].
3.  **Generation:** Модель имитирует экспертный стиль из примеров, гарантируя точность нормативов и стиля рекомендаций.

## 3. Гибкая отчетность (Template Engine)
**Проблема:** Сейчас PDF-отчет генерируется через хардкод HTML-строк в Python. Это затрудняет смену дизайна или добавление логотипов.

**Предложение:**
*   **Jinja2 Templates:** Вынести всю верстку отчетов в `.html` шаблоны.
*   **Брендирование:** Возможность загружать логотип компании и менять цветовые акценты через админку без пересборки бэкенда.

## 4. Качество кода и DX (Developer Experience)
**Проблема:** Медленный цикл «правка - билд - проверка».

**Предложение:**
*   **UV Migration:** Полный переход на `uv` для мгновенной установки зависимостей и детерминированных билдов.
*   **Docker Watch:** Внедрение `docker-compose watch`, чтобы изменения в коде подхватывались контейнерами без рестарта (HMR для фронта и reload для бэка).
*   **Gatekeeping:** Автоматический прогон `make check` перед каждым пушем, блокирующий билд при наличии линт-ошибок.

## 5. Визуализация и UI (Canvas 2.0)
**Проблема:** Статичные картинки с рамками не позволяют «провалиться» в детали.

**Предложение:**
*   **Interactive Overlays:** Использовать библиотеку типа `Konva.js` или `Fabric.js` для отрисовки дефектов. Это позволит зумить фото, скрывать/показывать разные типы дефектов слоями и кликать на рамку для быстрого перехода к описанию в списке.
